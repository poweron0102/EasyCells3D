01 Visão geral

O módulo de rede dá ao EasyCells3D um modelo cliente–servidor autoritativo: existe um servidor que detém a verdade sobre o jogo e N clientes que enviam intenções e recebem o estado de volta. Toda a comunicação é encapsulada de modo que, no código de jogo, você quase nunca lida com sockets — você lida com três abstrações de alto nível:

🛰️ RPC

Chamar um método decorado com @Rpc executa esse método nas outras máquinas em vez de localmente. Ideal para eventos: "atirar", "pulou", "spawnou".

📦 NetworkVariable

Um valor cujo .value é replicado automaticamente. Atribuir no dono propaga para todos; quem entra depois pede o estado atual via TCP.

🧭 NetworkTransform

Componente pronto que sincroniza posição, rotação e escala de um objeto pela rede, com filtro por eixo e supressão de pacotes redundantes.

Os arquivos centrais são NetworkTCP.py, NetworkUDP.py e o pacote NetworkComponents/ — que expõe publicamente NetworkManager, NetworkComponent, NetworkVariable, NetworkTransform, Rpc, SendTo e Protocol.

02 As 4 camadas

O módulo é estritamente estratificado. Cada camada só conhece a de baixo, o que mantém a complexidade de sockets isolada do código de jogo.

L4

NetworkTransform & seus componentes

Componentes de gameplay prontos. Usam RPCs e variáveis sem nunca tocar em sockets.

L3

@Rpc · NetworkComponent · NetworkVariable

Modelo de replicação: decorators, autoridade por dono, registro de objetos por identifier, relay do servidor.

L2

NetworkManager

Orquestrador. Abre TCP e UDP, decide para onde cada pacote vai e despacha os recebidos para o objeto certo no loop().

L1

NetworkServerTCP/UDP · NetworkClientTCP/UDP

Transporte cru: sockets, threads de aceitação/recepção, framing de bytes e (de)serialização com pickle.

O fluxo de uma chamada típica desce da L4 até a L1 no remetente, atravessa a rede, e sobe de volta da L1 até a L4 no destinatário, onde o método/variável correto é finalmente executado.

03 Camada de transporte L1

Quatro classes simétricas, uma por (papel × protocolo). Todas falam a mesma interface pública de quatro verbos, então as camadas acima as tratam de forma intercambiável.

MétodoServerClientO que faz
send(data, id) / send(data)✓✓Serializa e envia um objeto Python.
read()✓✓Não-bloqueante. Retorna o próximo objeto ou None se nada chegou.

TCP Stream confiável

TCP é um fluxo de bytes sem fronteiras de mensagem. Para saber onde uma mensagem termina, o módulo prefixa cada envio com um cabeçalho de 4 bytes (SIZE_SIZE) contendo o tamanho do payload em big-endian.

A leitura usa recv(..., MSG_PEEK) para espiar o buffer sem consumi-lo: se a mensagem inteira ainda não chegou, retorna None e tenta de novo no próximo frame — nunca trava o game loop.

UDP Datagrama rápido

UDP preserva fronteiras de pacote, então não há cabeçalho de tamanho — um recvfrom(65535) entrega uma mensagem inteira. Em troca, não há garantia de entrega nem de ordem.

Como UDP não tem "conexão", o servidor roda uma thread de recepção que lê todos os datagramas, identifica o cliente por (ip, port) e enfileira em um deque por cliente. O cliente se registra enviando um "HANDSHAKE" inicial.

Quadro de uma mensagem TCP

size (big-endian)
4 bytes · SIZE_SIZE
pickle.dumps(objeto) … →
N bytes · payload

Implementação interna — leitura não-bloqueante (TCP)

def read(self, client_id):
    s = self.clients[client_id]
    ready, _, _ = select.select([s], [], [], 0)   # há algo para ler?
    if not ready:
        return None
    head = s.recv(SIZE_SIZE, socket.MSG_PEEK)        # espia sem consumir
    if len(head) < SIZE_SIZE:
        return None                              # cabeçalho incompleto
    size = int.from_bytes(head, "big")
    if len(s.recv(size + SIZE_SIZE, socket.MSG_PEEK)) < size + SIZE_SIZE:
        return None                              # payload ainda chegando
    s.recv(SIZE_SIZE)                              # consome o header
    return pickle.loads(s.recv(size))           # consome e devolve o objeto
Convenção de IDs A lista clients começa em [None], então o índice 0 nunca é um cliente — ele representa o servidor. Os clientes reais ocupam os índices 1, 2, 3… e cada um recebe seu próprio ID assim que conecta (o servidor envia o número de volta no handshake).

04 NetworkManager L2

É um Component do engine e um singleton (NetworkManager.instance). Ele é a ponte entre o transporte e o modelo de replicação. Construir um já abre a rede: 

# No servidor
item.AddComponent(NetworkManager(ip="0.0.0.0", port=25765, is_server=True))

# No cliente — dispara connect_callback quando o TCP conecta
item.AddComponent(NetworkManager(ip="meu.servidor", port=25765, is_server=False,
                              connect_callback=lambda meu_id: print("entrei!", meu_id)))

Note que ele instancia os dois protocolos de uma vez: no servidor cria tcp_server + udp_server; no cliente, tcp_client + udp_client — ambos na mesma porta. O TCP é a conexão "mestre": é ele quem define o id do cliente e dispara o connect_callback do usuário.

Interface pública de envio

MétodoDescrição
send_to_server(packet, protocol)Cliente → servidor, no protocolo escolhido.
send_to_client(packet, id, protocol)Servidor → um cliente específico.
broadcast(packet, protocol)Servidor → todos os clientes.
call_rpc_on_client(id, metodo, *args)Servidor invoca um RPC num cliente; detecta o protocolo pela config do método.

Os três primeiros recebem um Protocol explícito e roteiam para tcp_* ou udp_*. As camadas acima sempre falam com o Manager — nunca diretamente com os sockets.

O coração: loop() e process_packet()

Como Component, o Manager tem seu loop() chamado todo frame. Ele drena ambos os transportes (TCP e UDP) de todos os clientes e entrega cada pacote ao process_packet, que faz o despacho final: 

def process_packet(data, sender_id):
    op_code, target_id, payload, args = data
    if op_code == OP_RPC:
        comp = NetworkComponent._active_components.get(target_id)
        if comp: comp.handle_incoming_rpc(payload, args, sender_id)
    elif op_code == OP_VAR:
        var = NetworkVariable._active_variables.get(target_id)
        if var: var.handle_network_update(payload, args, sender_id)

Repare no padrão de registro: objetos de rede se cadastram em dicionários estáticos (_active_components, _active_variables) indexados por identifier. O pacote carrega esse ID, e o Manager usa ele para reencontrar o objeto-alvo do outro lado da rede.

05 Formato do pacote

Acima do transporte, tudo trafega como uma tupla de 4 elementos. É o "envelope" comum a RPCs e variáveis:

( op_code , target_id , payload , args )

op_codeOP_RPC = 1 ou OP_VAR = 2 — diz ao receptor qual subsistema trata o pacote.
target_idO identifier do objeto destino. 0 (STATIC_NET_ID) = RPC estático/global.
payloadNome do método (RPC) ou a sub-operação VAR_SET/VAR_GET (variável).
argsTupla de argumentos, serializada junto pelo pickle.

Essa uniformidade é o que permite ao process_packet ser tão curto: ele lê o op_code, encontra o objeto pelo target_id e delega o resto.

06 Sistema de RPC L3

Um Remote Procedure Call é um método que, quando chamado localmente, na verdade executa nas outras máquinas. A mágica está no decorator @Rpc, que envolve o método: na primeira chamada ele não roda o corpo — ele empacota a chamada e a envia pela rede. Quando o pacote chega do outro lado, o módulo marca _is_executing_rpc = True e aí sim executa o corpo de verdade. 

class Player(NetworkComponent):

    @Rpc(send_to=SendTo.ALL, require_owner=True, protocol=Protocol.TCP)
    def disparar(self, direcao):
        spawn_projetil(direcao)        # roda em todos os peers

# Em qualquer lugar do gameplay:
player.disparar(Vec3(0, 0, -1))      # vira um pacote na rede, não roda local*

Os três parâmetros

send_to

Um SendTo que define o alcance: ALL, SERVER, CLIENTS, OWNER ou NOT_ME (todos menos quem enviou — clássico para sincronizar movimento).

require_owner

Se True, o servidor rejeita o RPC caso o remetente não seja o owner do objeto. É a checagem de autoridade — impede um cliente de comandar o personagem de outro.

protocol

Protocol.TCP (padrão, confiável) ou Protocol.UDP (rápido, para eventos frequentes e descartáveis como sincronizar transform).

Servidor como relay

Num modelo autoritativo os clientes não falam entre si. Quando o servidor recebe um RPC, ele primeiro valida a autoridade, executa localmente se preciso, e então retransmite (relay) para os outros clientes respeitando o send_to e o protocol originais. Veja o caminho de um RPC NOT_ME disparado por um cliente:

Fluxo de um RPC · send_to = NOT_ME

Cliente 1 (dono)
Servidor
Cliente 2
01 → send_rpc (OP_RPC, id, "disparar", args)
02 valida owner · executa local
03 → relay para todos menos o sender
04 handle_incoming_rpc → executa
* exceção importante Se você é o servidor e o send_to é ALL, o corpo do método roda imediatamente na sua máquina além de ser enviado — afinal o servidor também participa do "todos". Em qualquer outro caso, a chamada local apenas envia o pacote.

RPCs estáticos

Funções livres e @staticmethod decoradas com @Rpc também funcionam: elas usam a instância especial de identifier = 0 (STATIC_NET_ID) e são registradas em NetworkComponent._static_rpcs pelo nome. Servem para eventos globais que não pertencem a nenhum objeto específico (ex.: "iniciar partida").

07 NetworkVariable[T]

Onde RPCs sincronizam eventos, NetworkVariable sincroniza estado contínuo: vida, pontuação, nome do jogador. É genérica e expõe uma única propriedade pública, .value. Ler é local; escrever propaga. 

vida = NetworkVariable(100, identifier=42, owner=meu_id)

vida.value -= 10     # o setter envia OP_VAR/VAR_SET pela rede (sempre TCP)
print(vida.value)   # leitura puramente local

Variáveis usam sempre TCP — perder uma atualização de estado é inaceitável. A lógica de autoridade é a mesma dos RPCs: com require_owner=True, o servidor descarta um VAR_SET vindo de quem não é dono, e então faz relay do valor aceito para os demais.

Late-join: a sub-operação VAR_GET

Quando um cliente cria uma NetworkVariable que já existe no servidor, ele não conhece o valor atual. Por isso, no construtor, o cliente dispara um VAR_GET; o servidor responde com um VAR_SET direcionado só a ele, contendo o valor corrente. Assim quem entra no meio da partida recebe o estado certo.

Detalhe de implementação O registro _active_variables é um WeakValueDictionary: se a variável for coletada pelo garbage collector, ela some do registro automaticamente, sem vazar entradas mortas.

08 NetworkTransform L4

O melhor exemplo de tudo junto. É um NetworkComponent que mantém posição, rotação e escala de um objeto iguais em todos os peers. Você escolhe quais eixos sincronizar e a frequência: 

item.AddComponent(NetworkTransform(identifier=7, owner=meu_id,
                                 sync_frequency=0.015,   # ~66 Hz
                                 sync_scale_x=False))     # ignora escala em X

Como funciona por dentro

  • Só o dono envia. No init ele agenda uma task no scheduler; o loop de sync só roda se self.owner == NetworkManager.instance.id.
  • Serialização binária compacta. Em vez de pickle genérico, usa struct.pack para empacotar só os floats dos eixos habilitados — bem mais enxuto.
  • Supressão de redundância. A cada tick compara o pacote novo com last_sent; se nada mudou (ignorando o contador), não envia. Objeto parado não gera tráfego.
  • Transmite por UDP. O método de aplicação é @Rpc(send_to=NOT_ME, protocol=UDP) — perdeu um frame de posição? O próximo corrige. Velocidade > confiabilidade aqui.
  • Contador de sequência. Cada pacote leva um inteiro cont. Na recepção, se cont < self.cont o pacote é descartado — é assim que se rejeita um datagrama UDP que chegou fora de ordem.
async def sync(self):
    if self.owner == NetworkManager.instance.id:   # só o dono emite
        while True:
            data = self.serialize()
            if data[4:] != self.last_sent[4:]:   # mudou? (ignora o contador)
                self.last_sent = data
                self.sync_transform(data)         # RPC UDP → demais peers
            await self.game.scheduler.sleep(self.sync_frequency)

@Rpc(send_to=SendTo.NOT_ME, require_owner=True, protocol=Protocol.UDP)
def sync_transform(self, data):
    self.deserialize(data)                     # aplica nos outros (com checagem de cont)

Esse componente demonstra a regra geral do módulo: estado discreto e crítico → TCP + NetworkVariable; estado contínuo e tolerante a perda → UDP + RPC com contador de sequência.

09 Notas & armadilhas

Segurança · pickle Toda a (de)serialização usa pickle, que executa código arbitrário ao desempacotar dados maliciosos. É prático para um jogo entre amigos, mas nunca exponha o servidor à internet aberta sem trocar por um formato seguro. Trate todo peer como confiável.
Pareamento de IDs TCP/UDP TCP e UDP são conexões independentes. O módulo assume que o ID atribuído no UDP coincide com o do TCP porque a ordem de conexão tende a ser a mesma — mas isso é frágil sob concorrência. O TCP é sempre a fonte de verdade do id do cliente.
Tamanho do datagrama UDP entrega no máximo ~64 KB por pacote e sem fragmentação própria. Pacotes muito grandes (um NetworkTransform com muitos eixos cabe folgado, mas dados arbitrários não) devem ir por TCP.
Threads Aceitação de clientes (TCP) e recepção (UDP) rodam em threads daemon — encerram junto com o programa. O processamento dos pacotes, porém, acontece no loop() do game thread, então o seu código de jogo nunca corre risco de concorrência ao reagir a um RPC.
